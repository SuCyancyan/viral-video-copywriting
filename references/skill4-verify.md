# Skill 4：改稿（验证）

验证文案中所有数据、案例的真实性，给出修改建议，由用户决定是否采纳。

## 输入要求

| 输入项 | 来源 | 必填 |
|--------|------|------|
| 完整文案 | Skill 3 输出 | ✅ |
| 参考来源列表 | Skill 3 输出的来源标注 | ✅ |
| 验证重点 | 数据 / 案例 / 引用（默认全验） | ❌ |

## 核心原则

- **只给建议，不改稿**：最终决定权在用户
- **避免过度纠错**：聚焦事实性错误，不纠结措辞偏好
- **允许用户拒绝**：建议不合适，用户可以不采纳

## 执行流程

### 第1步：打开来源

逐一打开文案中标注的所有链接：

```
处理逻辑：
├── 链接可访问 → 进入第2步
├── 链接失效/付费墙 → 标记【来源不可访问】，建议用户手动核实
└── 链接不存在（404）→ 标记【来源已失效】，建议更换来源或删除该引用
```

### 第2步：摘要核查

阅读来源的摘要/开头部分：

```
判断逻辑：
├── 来源内容与文案表述一致 → ✅ 通过，进入第3步精确核查
├── 来源内容与文案表述相悖 → ❌ 标记为事实错误，给出修改建议
├── 来源内容部分支持 → ⚠️ 标记为部分偏差，说明偏差点
└── 来源内容未提及相关信息 → ⚠️ 进入第3步进一步搜索
```

### 第3步：关键词精确搜索

在来源全文中搜索文案提到的具体数据/案例：

```
搜索逻辑：
├── 找到完全匹配 → ✅ 通过
├── 找到近似匹配（数字略有差异）→ ⚠️ 标记偏差，给出来源原文数据
├── 完全找不到 → ❌ 标记为【来源中未找到该数据】，建议：
│   ├── A. 删除该数据
│   ├── B. 替换为来源中实际存在的数据
│   └── C. 寻找新的权威来源支撑
└── 来源提及方向但无具体数字 → ⚠️ 文案数字可能是推算，建议标注"约"或删除精确值
```

### 第4步：补充发现

核查过程中发现来源中有价值但文案未提及的信息：

```
补充规则：
├── 与文案主题直接相关 → 建议补充进文案
├── 能强化论点 → 建议补充并标注来源
└── 间接相关 → 仅告知用户，不做强制建议
```

## 防幻觉机制

```
三大红线：
  1. 不编造来源内容 → 看到什么写什么，看不到了就标注"无法验证"
  2. 不过度纠错 → 措辞偏好不算错误，数字不精确不等于造假
  3. 不强制修改 → 每条建议末尾都要写"由用户决定是否采纳"

灰色地带处理：
  - 数据四舍五入差异（如5000万 vs 4980万）→ 标注但不建议修改
  - 统计口径差异（如"用户数"vs"月活用户数"）→ 建议明确口径
  - 引用断章取义 → 明确标注原文完整语境，建议修正表述
```

## 注意事项

- 验证顺序：先打开所有链接确认可访问性，再逐条核查内容
- 如果来源是论文，重点看摘要+结论+关键数据图表即可，不必全文精读
- 如果来源是视频/播客，跳转时间戳定位关键信息
- 一条文案可能有多个数据点引用同一来源，每个数据点都要单独核查
